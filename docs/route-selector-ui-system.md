# Kathmandu route selector UI system

## Goal

The `/button` prototype explores a bus route selection flow for Kathmandu Valley where many routes overlap. It is intentionally sparse: a compact route app shell, two location selectors, one location popover, and a route rail.

1. Current location and final destination selection.
2. Multi-select route choice beneath the location pair.

The main UI invention is a location word cloud: a spatial, elastic surface where each place name sits near its real-world neighbors.

## Mechanism

The current implementation uses a topological cartogram for important Kathmandu Valley places. The source idea is still geographic projection:

```txt
x = (lng - minLng) / (maxLng - minLng) * 100
y = (maxLat - lat) / (maxLat - minLat) * 100
```

The inverted latitude keeps north visually higher on screen. For the prototype, those geographic positions are quantized into a non-overlapping cartogram surface. The visible picker is a smaller square window over that larger surface, similar to navigating an Apple Watch home screen.

## Tile sizing

Each word tile receives:

- A cartogram position derived from projected Kathmandu Valley coordinates.
- A standardized size generated by `tileMetrics`: compact, standard, wide, or hub.
- A standardized rounded rectangle size generated by `tileMetrics`: compact, standard, wide, or hub.

The `tileMetrics` heuristic uses only character length and neighbor count. Longer names become wider. Places with four or more neighbors become hub cells. Shape generation is intentionally disabled: every location is rendered as a simple rounded rectangle so the map reads cleanly and does not contain arbitrary polygon cutouts.

The edge reduction is intentional. It communicates that the surface can move, and the larger center area is where the user is expected to make precise choices.

## Interaction

- Clicking `Current` or `Destination` opens a small square popover directly below that selector.
- `Use GPS estimate` currently sets the current location to Putalisadak as a stand-in for browser GPS.
- Pointer movement uses an edge-scroll heuristic: the center of the picker stays steady, the outer 34% edge bands generate nonlinear intent, and holding the cursor closer to the absolute boundary continuously drifts the larger topological surface faster. The spring transition adds the elastic feel of the Apple Watch home screen.
- Clicking a word writes it into the active field.
- Routes are scored by direct coverage, endpoint coverage, and neighbor overlap, then shown as selectable route cards.

## Future production work

- Replace approximate coordinates with GTFS stop data or a maintained stop registry.
- Generate adjacency through Delaunay/Voronoi geometry rather than hand-authored neighbors.
- Add geolocation permission handling and reverse geocoding from GPS to nearest route stop.
- Add route transfer scoring when no single route covers both endpoints.
- Add keyboard search for accessibility and for users who already know their stop name.
